## §1 Writing Convention  {#s1-writing-convention}

본 문서 및 후속 wiki 관련 문서는 **AI-optimized** 형식을 따른다. 별도 style guide 작성하지 않음 — 이 절이 가이드 역할.

| 원칙 | 적용 |
|---|---|
| **YAML frontmatter** | 메타·결정·validator·scope·lineage 모두 머신 파싱 가능 |
| **Stable section anchors** | `## §N Title {#sN-slug}` — 외부 cross-reference 안정 |
| **Tables over prose** | 구조화 데이터는 항상 table (decision matrix, rule schema, validation) |
| **No narrative fluff** | 배경 서술·"이 문서는 ~다" 최소화; 결정·규칙·근거 중심 |
| **Rule IDs 영구 보존** | R1, R2, ... 개정 시 ID 유지, version 만 bump |
| **Anti-pattern IDs 영구 보존** | A1, A2, ... 동일 |
| **Validation IDs 영구 보존** | V-1, V-2, ... |
| **Phase IDs 영구 보존** | P1, P2, ... |
| **Cross-reference 형식** | 상대경로 + `#anchor` (예: `[§4 Rules](path#s4-rules)`) — stable link |
| **Single source of truth** | 중복 서술 금지; parent/child 관계는 frontmatter `parent` 로 명시 |
| **Compact revision log** | [§10 Revision Log](#s10-revision-log) — date / version / change 1줄 |
| **Language policy** | 한국어 (user-facing) + 영어 (technical terms, code, paths, keys) — `MEMORY_GOVERNANCE.md` §2 준수 |

## §2 Decisions  {#s2-decisions}

| ID | 결정 | 근거 | 확정일 |
|---|---|---|---|
| **D1** | wiki 위치 = `ai-workflow/wiki/` (Runtime layer) | ADR-001 §3 의 Runtime layer 안에 흡수. Project docs 와 분리 유지. `docs/wiki/` 는 Project docs 침범, `workflow-source/wiki/` 는 Source layer 침범 → 둘 다 거부 | 2026-06-12 |
| **D2** | wiki 추적 = **git 추적** (memory/ 는 gitignore 유지) | 분산·merge·history 의 전제. memory (volatile session state) 와 wiki (durable knowledge) 를 별도 정책으로 운용. ADR-001 의 spirit 보존 (layer 분리) | 2026-06-12 |
| **D3** | ingest 트리거 = **per-session** (session 종료 시) | M1 모드. M3 (release-freeze) 는 자동 파생 (release 스냅샷이 wiki/ 까지 포함) | 2026-06-12 |

### §2.1 Layer 분리 정책  {#s2-1-layer-separation-policy}

```
ai-workflow/
├── memory/        # gitignore 유지 (volatile session state, 현행 유지)
│   └── state.json   # ← git 추적 캐시만 예외 (현행)
└── wiki/          # git 추적 (durable knowledge, 분산 가능) — 본 plan 의 신규 scope
```

**불변식**: `memory/` 와 `wiki/` 는 디렉토리·정책·lifecycle 모두 분리. cross-link 만 허용 (한쪽에서 다른쪽 파일 read OK, write 는 자기 scope 만).

## §3 Architecture  {#s3-architecture}

### §3.1 File tree  {#s3-1-file-tree}

```
ai-workflow/
├── memory/                    # gitignore — workflow state
└── wiki/                      # git 추적 — knowledge
    ├── index.md               # 페이지 카탈로그 (anchor 기반, R4)
    ├── log.md                 # ingest/query 작업 로그 (append-only)
    ├── SCHEMA.md              # 운영 헌법 (페이지 타입·frontmatter·workflow)
    ├── entities/              # Component / Service / API / Person / System
    ├── concepts/              # MCP / Skill / Orchestrator / Pydantic / CRDT 등
    ├── decisions/             # ADR 형식
    ├── patterns/              # 재사용 패턴 (harness overlay, memory write merge 등)
    └── queries/               # file-back 답변 (구문·분석·합성 결과)
```

### §3.2 Layer mapping (ADR-001 ↔ wiki)  {#s3-2-layer-mapping}

| ADR-001 Layer | 위치 | wiki 요소 | 비고 |
|---|---|---|---|
| Source | `workflow-source/templates/wiki/` | wiki/SCHEMA.md 템플릿 (부트스트랩용) | 신규 |
| **Runtime** | `ai-workflow/wiki/` | **실제 운영 wiki** | 본 plan 의 primary scope |
| Project Docs | (해당 없음) | wiki 는 Project docs 아님 | 분리 유지 — wiki 는 workflow state 의 일종 |

### §3.3 .gitignore 정책  {#s3-3-gitignore-policy}

`.gitignore` 변경 필요 (ADR-001 carve-out):

```gitignore
# 기존 (유지)
ai-workflow/memory/
ai-workflow/mcp_servers/
ai-workflow/core/
ai-workflow/skills/
...

# 추가 (변경)
!ai-workflow/wiki/             # wiki 는 git 추적 (D2)
!ai-workflow/wiki/**/*.md
```

### §3.4 Ingest → Query → Lint 흐름  {#s3-4-ingest-query-lint-flow}

```
session start
  └─► session-start ──► wiki/index.md (load) ──► 관련 페이지 (lazy load)

session work
  └─► (수동 ingest hint) backlog-update → 후보 페이지 플래그

session end
  └─► wiki-ingest (자동)
       ├─ handoff.md + work_backlog.md + latest backlog 읽기
       ├─ 언급된 entity/concept/pattern 식별 (5~15)
       ├─ 관련 페이지 multi-page 갱신 (atomic, R2)
       ├─ [CONTRADICTION] 발견 시 양쪽 페이지 명시 (R5)
       ├─ index.md anchor 추가/갱신 (R4)
       ├─ log.md append (## [YYYY-MM-DD] ingest | <summary>)
       └─► 1 commit (wiki-ingest: <session-summary>)

periodic (release / weekly / manual)
  └─► wiki-lint (R7 + §5 안티패턴)
       ├─ 모순 / 스테일 / 고아 / 누락 검사
       ├─ index.md anchor 무결성
       └─► lint report → 다음 ingest 의 hint
```

## §4 Rules  {#s4-rules}

각 규칙은 **영구 ID** 보유. 개정 시 ID 유지, version 만 bump.

### R1: Wiki Location  {#r1-wiki-location}

| 필드 | 값 |
|---|---|
| Statement | wiki 는 `ai-workflow/wiki/` 에 둔다. `docs/wiki/`, `workflow-source/wiki/`, `.wiki/`, `wiki/` (root) 등 다른 위치 사용 금지 |
| Why | Runtime layer (ADR-001 §3) 단일화로 부트스트랩·export·harness overlay 정책 일관성 유지. Project docs 침범 방지 |
| Implementation | `bootstrap_lib --enable-wiki` 옵션 시 디렉토리·SCHEMA·index.md·log.md 생성. 6개 harness overlay 에 동일 path emit |
| Validator | V-1 (R1 참조) |

**Validation**:
- `[ ]` `ai-workflow/wiki/index.md` 존재
- `[ ]` `docs/wiki/`, `.wiki/`, `workflow-source/wiki/` 등 중복 디렉토리 0건
- `[ ]` root `wiki/` 디렉토리 0건 (있을 경우 R1 violation)

### R2: Page Atomicity  {#r2-page-atomicity}

| 필드 | 값 |
|---|---|
| Statement | 1 commit = 1 ingest (5~15 페이지 동시 갱신). 페이지 = primary record atomic unit. 페이지 내부 문단 단위 write 금지 |
| Why | 문단 단위 write = prose merge 함정 (CRDT/LWW/OT 어느 것도 의미적 모순 못 잡음). 페이지 단위 = 검토·rollback 가능 |
| Implementation | `wiki-ingest` skill: session 종료 시 handoff+backlog 분석 → 관련 페이지 식별 → 단일 commit 으로 동시 갱신. 페이지 frontmatter `last_ingested_from` (source handoff path) 필수 |
| Validator | V-2 (R2 참조) |

**Validation**:
- `[ ]` commit 단위: `git log --grep="^wiki-ingest:"` 파싱 → 1 commit = 1 ingest 확인
- `[ ]` 1 commit 내 touched pages = 5~15 범위 (0~4, 16+ 는 warning)
- `[ ]` frontmatter `last_ingested_from` 없는 페이지 = 0건

### R3: Pull-Before-Push  {#r3-pull-before-push}

| 필드 | 값 |
|---|---|
| Statement | wiki push 직전 `git fetch && git rebase origin/main` 필수. 동시 ingest 검출 시 atomic counter 로 serialize |
| Why | 옵시디언-git 의 ref lock cascade 실패 패턴 차단 ([known issue](https://github.com/denolehov/obsidian-git)). 자동 push / force push 금지 |
| Implementation | `wiki-ingest` skill sync protocol: (1) fetch → (2) rebase `-- ai-workflow/wiki/` → (3) 충돌 시 LLM review 큐 등록 (자동 resolve 안 함) → (4) 통과 시 ingest commit → (5) atomic counter (`ai-workflow/wiki/.ingest_lock`, gitignore) 동시 ingest 검출 시 block |
| Validator | V-3 (R3 참조) |

**Validation**:
- `[ ]` rebase 없이 push 한 commit = 0건 (CI hook 검증)
- `[ ]` `.ingest_lock` 경합 시 reject 로그 존재
- `[ ]` force push 시도 = 0건 (reflog 검사)

### R4: Index Structure  {#r4-index-structure}

| 필드 | 값 |
|---|---|
| Statement | `wiki/index.md` 는 **anchor 기반 구조화**, 자유 산문 금지. 각 entry = 파일 링크 + anchor ID |
| Why | 모든 commitor 가 쓰는 단일 파일 = 영구 충돌 지점. anchor 단위 merge = 충돌 격리. append-only entry 는 conflict 0 |
| Schema | `### [[<path>]] {#<anchor-id>}` (예: `### [[concepts/mcp-transport]] {#mcp-transport}`) |
| Validator | V-4 (R4 참조) |

**Schema example**:

```markdown
# Master Knowledge Index

## Concepts
### [[concepts/mcp-transport]] {#mcp-transport}
### [[concepts/session-continuity]] {#session-continuity}

## Decisions
### [[decisions/ADR-004-wiki-layer]] {#adr-004}

## Patterns
### [[patterns/harness-overlay]] {#harness-overlay}
```

**Conflict surface**: anchor ID 단위로 한정. 새 entry append = conflict 0건. 기존 entry 수정 시 section-conflict 발생 → [R5 additive](#r5-additive-merge) + [R7 merge extension](#r7-merge-resolution-extension) 적용.

**Validation**:
- `[ ]` index.md 에 anchor 없는 섹션 = 0건
- `[ ]` index anchor ID 중복 = 0건
- `[ ]` 링크 깨진 entry (target 파일 없음) = 0건

### R5: Additive Merge  {#r5-additive-merge}

| 필드 | 값 |
|---|---|
| Statement | 충돌 시 **양쪽 결합**, 폐기 금지. LLM reviewer 가 canonical 선택 |
| Why | Letta MemFS 검증 패턴. 정보 손실 방지. reasoning 보존 |
| Implementation | `merge-doc-reconcile` 위키 확장 사용. 충돌 결과 = 두 버전 모두 보존 (alternative 섹션). LLM reviewer 가 canonical 표시 + `[REVIEWED by {agent} at {date}]` 태그 부착 |
| Validator | V-5 (R5 참조) |

**Conflict result schema**:

```markdown
## {heading}

[version A]
{...content A...}

---

[version B]
{...content B...}

[REVIEWED by {agent-id} at {YYYY-MM-DD}] — 선택: A / B / 통합본
```

**Validation**:
- `[ ]` 충돌 commit 에 양쪽 보존 안 한 경우 = lint error
- `[ ]` `[REVIEWED]` 태그 없는 충돌 결과 = 0건 (24h 이내)
- `[ ]` 자동 resolve 된 semantic-conflict = 0건 ([R7](#r7-merge-resolution-extension) 위배)

### R6: Topic-Branch Mode  {#r6-topic-branch-mode}

| 필드 | 값 |
|---|---|
| Statement | 큰 탐색 phase (6+ 페이지 분량, 새 패러다임 실험, harness-specific 위키 작업) 만 `wiki/topic/<name>` 브랜치 사용. PR promote 로 main 통합 |
| Why | 일반 작업은 R1~R5 로 충분. topic branch 는 격리·검토·롤백 안전망 |
| Naming | `wiki/topic/<kebab-case-descriptive-name>` (예: `wiki/topic/multi-agent-mem-pattern`, `wiki/topic/harness-codex-wiki-overlay`) |
| Validator | V-6 (R6 참조) |

**Lifecycle**:

```
main (default branch)
  └── wiki/topic/<name>   (격리 작업, 자유 commit)
       └── PR (additive merge + LLM review 필수)
            └── main promote
```

**Promotion criteria**:
- `[ ]` PR description 에 탐색 목적·결과·영향 페이지 명시
- `[ ]` LLM review 통과 (semantic-conflict 0)
- `[ ]` main 으로 merge 시 R5 additive resolution 적용
- `[ ]` topic branch 가 main 에 merge 된 후 close (stale branch = 7일 후 lint warning)

**Validation**:
- `[ ]` topic branch → main PR 에 review 기록 = 1건 이상
- `[ ]` 직접 push to main (topic branch 우회) = 0건
- `[ ]` main 직접 ingest (topic branch 거치지 않은 큰 변경) = warning

### R7: Merge-Resolution Extension  {#r7-merge-resolution-extension}

| 필드 | 값 |
|---|---|
| Statement | `merge-doc-reconcile` skill 을 wiki 전용으로 확장. conflict type 별 resolution 명시. read-only default 유지, `--apply` 는 LLM 승인 후 한정 |
| Why | 기존 `merge-doc-reconcile` 는 일반 문서 read-only. wiki 는 의미적 모순 감지·additive write 필요 |
| Validator | V-7 (R7 참조) |

**Conflict type 분류**:

| Type | 설명 | Resolution |
|---|---|---|
| `line-conflict` | 동일 라인 동시 수정 | reconcile-text (word-level OT) |
| `section-conflict` | 동일 섹션 동시 수정 | additive (R5) |
| `semantic-conflict` | 의미적 모순 (두 설명이 모순) | LLM review 필수 (자동 resolve 0) |
| `index-conflict` | index.md anchor 충돌 | manual review 필수 |

**Mode 정책**:
- default = read-only (기존과 동일)
- `--apply` mode = LLM 승인 + `--confirm-llm-review` 플래그 시에만 write
- `semantic-conflict` 분류 = LLM review 큐에만 등록, 자동 resolve 0건

**Validation**:
- `[ ]` `--apply` 호출 시 `confirm-llm-review` 없으면 reject
- `[ ]` semantic-conflict 분류된 파일은 LLM review 큐 등록 외 자동 resolve = 0건
- `[ ]` line-conflict 가 reconcile-text (L4 OT) 거치지 않은 경우 = 0건

### R8: Memory Raw Freeze  {#r8-memory-raw-freeze}

| Field | Value |
|---|---|
| Statement | session 종료 시 `memory/active/` → `memory/archive/YYYY-MM-DD/` atomic move + `.frozen` 마커. freeze 후 read-only |
| Why | Karpathy "immutable raw" 요건을 시간축으로 만족. 다음 세션 agent 가 raw 보존 |
| Implementation | `memory/active/_freeze.py` 신규 또는 `session-end` 스킬 통합 — atomic rename + state.json 일치 검증 + log.md append |
| Validator | V-R8 |

### R9: Raw Source of Truth  {#r9-raw-source-of-truth}

| Field | Value |
|---|---|
| Statement | wiki-ingest 는 `memory/archive/` 만 ingest. `memory/active/` 절대 ingest 안 함 |
| Why | active 는 mutable, freeze 전이므로 ingest source 부적합. archive 만 immutable raw |
| Implementation | `wiki-ingest` 스킬 source filter: `archive/YYYY-MM-DD/` 만 허용. active/ 경로 reject |
| Validator | V-R9 |

### R10: Freeze Lint  {#r10-freeze-lint}

| Field | Value |
|---|---|
| Statement | freeze 트랜지션 시 5개 파일 (handoff/daily backlog/tasks/state/log) 무결성 + state.json 일치 자동 검증 |
| Why | 부분 freeze 시 raw 깨짐 → wiki-ingest source 신뢰성 붕괴 |
| Implementation | `check_memory_freeze_lint.py` (신규) — freeze 직전 5 파일 read + state.json path 일치 + 필수 frontmatter 검증 |
| Validator | V-R10 |

## §5 Anti-Patterns  {#s5-anti-patterns}

| ID | 안티패턴 | 위험 | 대안 |
|---|---|---|---|
| **A1** | 페이지 분할해서 동시 작업 (wiki A1·A2 페이지 분담, 같은 페이지 분단 수정) | merge 시 backlink graph 깨짐, 의미적 모순, atomicity (R2) 위반 | 단일 페이지 = 단일 owner; 동시 작업 필요 시 topic branch (R6) |
| **A2** | 자동 push (CI/hook/agent 가 ingest commit 자동 push) | 옵시디언-git ref lock cascade 실패 패턴, 동시 ingest data loss | R3 sync protocol 강제; 자동 push hook = lint error |
| **A3** | `index.md` 자유 산문 작성 / anchor 미부착 섹션 | 모든 commitor 가 같은 파일 → 영구 충돌 지점, R4 위반 | R4 anchor 기반 구조화 강제; lint 가 anchor 미부착 섹션 발견 시 reject |
| **A4** | wiki merge 결과를 사람/LLM 안 보고 자동 accept | 의미적 모순이 "그럴듯한 합성"으로 가려짐; R5·R7 위반 | R5 additive + R7 semantic-conflict LLM review 의무; 자동 accept flag 사용 = 0 |
| **A5** | `memory/active/` 를 wiki-ingest source 로 사용 (R9 위반) | mutable raw → 모순/부분 데이터 → wiki 페이지 신뢰성 붕괴 | archive/ 만 ingest. R9 source filter |
| **A6** | freeze 없이 archive/ 직접 write (R8 위반) | raw immutability 위반. provenance 추적 불가 | active/ 에서 작업 후 freeze 트랜지션 |

## §9 ADR-004 Draft  {#s9-adr-004-draft}

> **ADR-004: LLM Wiki Layer 도입**
>
> | 필드 | 값 |
> |---|---|
> | Status | proposed |
> | Date | 2026-06-12 |
> | Deciders | Sisyphus, maintainer |
> | Parent | [.omo/plans/llm-wiki-convergence-design.md](llm-wiki-convergence-design.md#s6) |
>
> **Context**
>
> Karpathy LLM Wiki (2026-04) 패턴 분석 결과, 우리 `ai-workflow/memory/` 는 운영 state 로는 우수하나 지식 위키로는 빈약. 지식 누적·합성·재사용을 위한 별도 layer 필요. `memory/` 의 gitignore 정책과 의도적으로 분리된 durable knowledge layer 가 요구됨.
>
> **Decision**
>
> `ai-workflow/wiki/` 신설. git 추적. ingest-first (per-session) + release-freeze (자동 파생). 7개 규칙 (R1~R7) + 4개 안티패턴 (A1~A4) + 8개 검증 (V-1~V-8) 적용.
>
> **Consequences**
>
> Positive:
> - (+) 지식 누적·합성·재사용 가능 (RAG 대비 ~70x 토큰 효율, 500KB 이내)
> - (+) ADR-001 3-Layer 정신 유지 (Runtime layer 안에 흡수, project docs 침범 안 함)
> - (+) 기존 인프라 활용 (bootstrap_lib, contract_v1, 6 harness overlay, merge-doc-reconcile)
> - (+) LLM Wiki = LLM-friendly 문서 작성 방식으로 agent 의 wiki 활용도 극대화
>
> Negative:
> - (-) 위키 운영 헌법 (SCHEMA) · lint 스킬 · ingest 스킬 등 신규 컴포넌트 필요
> - (-) `merge-doc-reconcile` 위키 확장 (R7) 필요
> - (-) `.gitignore` carve-out 정책 변경 (D2)
> - (-) 6 harness overlay wiki/ 동기화 작업 (P4)
>
> **Alternatives considered**
>
> | Option | Reason rejected |
> |---|---|
> | 메모리 대체 (방향 B, `.omo/plans/llm-wiki-convergence-design.md` §3) | 11개 스킬 + 6 하네스 재설계 필요. 위험 대비 효과 불명 |
> | Federated Wiki (Ward Cunningham) | federation discovery 가 scale 안 됨 (2018 archived). 우리 1인 dev 환경엔 과설계 |
> | CRDT (Yjs, Automerge) | tombstone overhead + 의미적 모순 미감지 + 1인 dev 에선 과설계 |
> | `docs/wiki/` (Project docs) | ADR-001 의 Runtime/Project docs 경계 침범. bootstrap/exclude 정책 충돌 |
> | `workflow-source/wiki/` (Source) | Source layer 는 템플릿·거버넌스 정의만. 운영 상태 불가 |
