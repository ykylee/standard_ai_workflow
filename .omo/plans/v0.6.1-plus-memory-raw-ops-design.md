## §1 Writing Convention  {#s1-writing-convention}

본 문서는 [`.omo/plans/v0.5.11-plus-llm-wiki-distributed-rules.md#s1-writing-convention`](v0.5.11-plus-llm-wiki-distributed-rules.md#s1-writing-convention) 과 **동일한 AI-optimized 컨벤션** 을 따른다 — 별도 style guide 작성하지 않음. 이 절은 single source of truth 역할의 wiki plan §1 을 reference 한다.

| 원칙 | 적용 |
|---|---|
| **YAML frontmatter** | 메타·결정·validator·scope·lineage 모두 머신 파싱 가능 |
| **Stable section anchors** | `## §N Title {#sN-slug}` — 외부 cross-reference 안정 |
| **Tables over prose** | 구조화 데이터는 항상 table (decision matrix, rule schema, validation) |
| **No narrative fluff** | 배경 서술·"이 문서는 ~다" 최소화; 결정·규칙·근거 중심 |
| **Rule IDs 영구 보존** | **R8, R9, R10** — 본 plan 신규. wiki plan 의 R1~R7 와 통합 운영. 개정 시 ID 유지, version 만 bump |
| **Anti-pattern IDs 영구 보존** | **A5, A6, A7, A8, A9** — 본 plan 신규. wiki plan 의 A1~A4 와 통합 운영 |
| **Validation IDs 영구 보존** | **V-R8, V-R9, V-R10, V-T1a/b/c/d, V-T2a/b, V-T3a/b/c** — 본 plan 신규 |
| **Phase IDs 영구 보존** | **P1.5, P2, P2.5, P3, P4** — wiki plan 의 P1~P4 와 별도 시퀀스 |
| **Cross-reference 형식** | 상대경로 + `#anchor` — stable link |
| **Single source of truth** | wiki plan 과 본 plan 의 중복 서술 금지. wiki plan 의 R1~R7 / A1~A4 / V-1~V-8 은 wiki plan 만 source. 본 plan 의 R8~R10 / A5~A9 / V-R*·V-T* 는 본 plan 만 source |
| **Compact revision log** | [§11 Revision Log](#s11-revision-log) — date / version / change 1줄 |
| **Language policy** | 한국어 (user-facing) + 영어 (technical terms, code, paths, keys) — `MEMORY_GOVERNANCE.md` §1 준수 |

## §3 Memory-as-Raw Architecture  {#s3-memory-as-raw-architecture}

### §3.1 Lifecycle diagram  {#s3-1-lifecycle-diagram}

```
   ┌─────────────┐
   │  session n  │ (활성 세션, 작업 중)
   └──────┬──────┘
          │ write/read
          ▼
   ┌─────────────────────┐
   │  memory/active/     │  ◄── mutable (session 진행 중 자유 갱신)
   │  ├── handoff.md     │
   │  ├── work_backlog.md│
   │  ├── backlog/       │
   │  ├── state.json     │
   │  └── log.md         │
   └──────────┬──────────┘
              │ session-end (D8)
              │ atomic rename + .frozen 마커 (R8)
              ▼
   ┌─────────────────────────────┐
   │  memory/archive/YYYY-MM-DD/ │  ◄── immutable raw (per-session, D9)
   │  ├── handoff.md             │
   │  ├── work_backlog.md        │
   │  ├── backlog/               │
   │  ├── state.json             │
   │  ├── log.md                 │
   │  └── .frozen                │  (마커 파일: freeze 시각·trigger 기록)
   └──────────────┬──────────────┘
                  │ wiki-ingest (R2, R9)
                  │ source = archive/ only
                  ▼
   ┌─────────────────────────────┐
   │  ai-workflow/wiki/          │  ◄── compiled knowledge (git 추적)
   │  ├── index.md               │  (wiki plan R4 anchor 기반)
   │  ├── log.md                 │
   │  ├── concepts/  decisions/  │
   │  ├── patterns/  entities/   │
   │  └── queries/               │
   └─────────────────────────────┘
                  │ release (deep freeze)
                  ▼
   ┌─────────────────────────────┐
   │  memory/release/v0.X.Y/     │  ◄── deep freeze (release 스냅샷)
   │  ├── archive/ (전체)        │
   │  ├── wiki/   (전체)         │
   │  └── MANIFEST.json          │
   └─────────────────────────────┘
```

### §3.2 Layer mapping  {#s3-2-layer-mapping}

| Karpathy LLM Wiki | 우리 매핑 | 정책 | 비고 |
|---|---|---|---|
| `raw/` (immutable) | `memory/archive/YYYY-MM-DD/` | **immutable per-session** (D9). freeze 후 read-only, `.frozen` 마커 부착 | Karpathy 의 "raw" = 시간축 freeze 후 1:1 매핑 |
| `wiki/` (compiled) | `ai-workflow/wiki/` | git 추적. wiki plan R1~R7 (위 분산 규칙) | 본 plan 의 R9 가 ingest source = `archive/` 만 허용 |
| `SCHEMA.md` (= 헌법) | `MEMORY_GOVERNANCE.md` (확장) + `ai-workflow/wiki/SCHEMA.md` | 본 plan 의 §4 R8~R10 + 위 plan 의 §4 R1~R7 의 통합 헌법 | memory + wiki 운영 헌법 단일화 |
| `index.md` (카탈로그) | `ai-workflow/wiki/index.md` | wiki plan R4 (anchor 기반) | 본 plan D6 (`work_backlog.md` 도 anchor 전환) 와 결합 |
| `log.md` (작업 로그) | `memory/log.md` (신규, D5) + `ai-workflow/wiki/log.md` | 양쪽 append-only. ingest/query/freeze 이벤트 기록 | memory log 는 gitignore, wiki log 는 git 추적 |
| **Ingest 운영** | `wiki-ingest` (wiki plan) | source = `archive/` only (R9) | 본 plan P2.5 에서 source filter 추가 |
| **Query 운영** | `session-start` 강화 + `memory-query` (T2) | index-based retrieval | 본 plan P3 |
| **Lint 운영** | `memory-lint` (T1) + `wiki-lint` (wiki plan) | 4종 (contradiction/stale/orphan/missing) | 본 plan P2 |
| **File-back 답변** | `memory/queries/` 또는 `ai-workflow/wiki/queries/` | 30줄 초과 시 page 화 | v0.6.3+ planned |

### §3.3 .gitignore 정책  {#s3-3-gitignore-policy}

`.gitignore` 변경 (ADR-001 정책 유지 + raw lifecycle 표지 추가):

```gitignore
# 기존 (유지)
ai-workflow/memory/                    # 메모리 전체 gitignore (ADR-001 §3 Runtime layer)
ai-workflow/mcp_servers/
ai-workflow/core/
ai-workflow/skills/
...

# 추가 (변경 없음, 단 raw 표지 추가)
!ai-workflow/wiki/                     # wiki 는 git 추적 (위 plan D2)
!ai-workflow/wiki/**/*.md

# raw lifecycle 의미적 표지 (D5·D7)
# archive/ 도 gitignore 유지하되, .frozen 마커 파일이 freeze 시점을 표시
# → git status 에 .frozen 변화 감지 → audit 가능
```

| 경로 | git 추적 | 비고 |
|---|---|---|
| `ai-workflow/memory/active/` | ❌ gitignore | mutable (session 진행 중) |
| `ai-workflow/memory/archive/YYYY-MM-DD/` | ❌ gitignore | immutable raw, `.frozen` 마커로 표지 |
| `ai-workflow/memory/log.md` | ❌ gitignore | append-only 작업 로그 (D5) |
| `ai-workflow/memory/state.json` | ⚠️ selective (cache) | 현행 유지 — `MEMORY_GOVERNANCE.md` §2 |
| `ai-workflow/memory/release/v0.X.Y/` | ❌ gitignore | deep freeze snapshot, MANIFEST.json 포함 |
| `ai-workflow/wiki/` | ✅ git 추적 | 위 plan D2 |

**불변식**: `memory/archive/` 는 gitignore 지만, `.frozen` 마커는 **state 외부 audit log** (예: `~/.workflow/memory_audit.log`) 에 동시 기록하여 추후 provenance 검증 가능. 단, raw immutability 자체는 파일시스템 ACL + lint (R10) 로 강제.

## §4 Rules (R8~R10, new rules extending wiki plan)  {#s4-rules-r8-r10-new}

각 규칙은 **영구 ID** 보유. 본 plan 의 R8~R10 은 wiki plan 의 R1~R7 과 통합 운영 (D10). 개정 시 ID 유지, version 만 bump.

### R8: Memory Raw Freeze  {#r8-memory-raw-freeze}

| 필드 | 값 |
|---|---|
| Statement | session 종료 시 `memory/active/` → `memory/archive/YYYY-MM-DD/` atomic move + `.frozen` 마커 생성. freeze 후 read-only |
| Why | Karpathy "immutable raw" 요건을 시간축으로 만족. 다음 세션 agent 의 raw 보존. wiki-ingest 가 신뢰할 수 있는 source 경계 (R9 와 결합) |
| Implementation | `memory/active/_freeze.py` 신규 스킬 (또는 `session-end` 스킬에 통합) — (1) atomic rename `active/` → `archive/YYYY-MM-DD/`, (2) `.frozen` 마커 파일 작성 (JSON: `{frozen_at, session_id, trigger, file_count}`), (3) state.json path 일치 검증 (R10), (4) `memory/log.md` append `## [YYYY-MM-DD] freeze | <summary>` (D5), (5) 5 파일 무결성 검증 통과 후에만 commit. `session-end` skill 또는 wiki-ingest 후속 hook 으로 trigger (D8) |
| Validator | V-R8 (R8 참조) |

**Validation**:
- `[ ]` `memory/archive/YYYY-MM-DD/.frozen` 존재 + JSON 스키마 일치
- `[ ]` freeze 후 `memory/active/` 비어있음 (또는 다음 세션용 빈 디렉토리만)
- `[ ]` `memory/log.md` 에 `## [YYYY-MM-DD] freeze` 행 존재
- `[ ]` atomic rename 중간 실패 시 archive/ 부분 파일 0건 (전부 또는 전무)

### R9: Raw Source of Truth  {#r9-raw-source-of-truth}

| 필드 | 값 |
|---|---|
| Statement | wiki-ingest 는 `memory/archive/` 만 ingest. `memory/active/` 절대 ingest 안 함 (일관성 깨짐) |
| Why | active 는 mutable, freeze 전이므로 ingest source 로 부적합. archive 만 immutable raw. ingest 중 active 가 갱신되면 wiki 와 raw 간 의미적 모순 발생 |
| Implementation | `wiki-ingest` skill 에 source filter 추가: `archive/YYYY-MM-DD/` 만 허용. `active/` 경로 발견 시 reject + lint error. `wiki-ingest` 호출 시 명시적 `archive_path` argument 필수 (default = `archive/`) |
| Validator | V-R9 |

**Validation**:
- `[ ]` `wiki-ingest` 호출 source path = `archive/...` 만 (0건 reject)
- `[ ]` ingest commit 의 source citation 에 `archive/YYYY-MM-DD/` 명시
- `[ ]` `active/` 경로 포함 wiki commit = 0건 (CI hook)

### R10: Freeze Lint  {#r10-freeze-lint}

| 필드 | 값 |
|---|---|
| Statement | freeze 트랜지션 시 5개 파일 (handoff/daily backlog/tasks/state/log) 무결성 + state.json 일치 자동 검증 |
| Why | 부분 freeze 시 raw 깨짐 → wiki-ingest source 신뢰성 붕괴. freeze 가 atomic 이라도 이전 상태의 corruption 을 detect 하지 못하면 의미 없음 |
| Implementation | `check_memory_freeze_lint.py` (신규) — freeze 직전/직후 5 파일 read + state.json path 일치 + 필수 frontmatter 검증 (`TASK-XXX` 의 `id/status/created_at`, `backlog/YYYY-MM-DD.md` 의 `Purpose/Status/Updated`, handoff 의 `Branch/Updated` 섹션, log.md 의 append-only 정합). 통과 시에만 freeze 진행 |
| Validator | V-R10 |

**Validation**:
- `[ ]` 5 파일 모두 존재 + 필수 frontmatter/section 일치
- `[ ]` `state.json.session_path` = `archive/YYYY-MM-DD/` 일치
- `[ ]` `backlog/YYYY-MM-DD.md` 와 `backlog/tasks/*.md` cross-link 정합 (link 깨진 TASK-XXX = 0)
- `[ ]` `log.md` append 시 timestamp 단조증가 (역전 = 0건)

## §5 Memory Operations (T1~T3)  {#s5-memory-operations-t1-t3}

3종의 신규 ops 가 wiki plan 의 `wiki-ingest`/`wiki-query`/`wiki-lint` 와 1:1 매핑. 단 source 차이 (memory raw → wiki compiled).

### T1: Memory Lint  {#t1-memory-lint}

| Type | Check | Implementation | Validator |
|---|---|---|---|
| **contradiction** | handoff `next_actions` vs `backlog/YYYY-MM-DD.md` 의 `done` 항목 불일치. handoff 가 "next = X" 인데 backlog 의 최신 일자에 X done 표시 = 모순 | `check_memory_lint.py` 신규 — handoff read + 최신 backlog read + diff | V-T1a |
| **stale** | 90일+ 미갱신 (archive 외). `state.json.last_ingested` 또는 파일 mtime 기준 | 동상 | V-T1b |
| **orphan** | inbound link 0 (`tasks/*.md`). 다른 파일에서 TASK-XXX 인용 없음 | 동상 — 모든 `.md` 에서 grep `TASK-XXX` | V-T1c |
| **missing** | TASK-XXX 언급 (handoff/backlog/work_backlog) but no `backlog/tasks/TASK-XXX.md` 파일 | 동상 | V-T1d |

**Validation**:
- `[ ]` T1 4종 모두 `check_memory_lint*.py` 통과 (CI hook)
- `[ ]` contradiction 0건 (release gate)
- `[ ]` stale 90일+ 페이지 0건 (warning)
- `[ ]` orphan / missing = 0건 (error)

### T2: Memory Query (index upgrade)  {#t2-memory-query}

| Step | 현행 | 개선 (T2) | 효과 |
|---|---|---|---|
| session-start 시 read | `state.json` + `session_handoff.md` + `work_backlog.md` linear read (3~4 파일 full read) | **anchor-based index** — `work_backlog.md` 의 `### [[TASK-XXX]] {#task-XXX}` anchor 만 로드, 필요 시점 lazy load | **50%+ 시간 단축** 목표. 컨텍스트 윈도우 절약 |
| 관련 task 검색 | grep full-text | anchor ID 기반 lookup (`#task-XXX` direct link) | O(1) retrieve |
| 우선순위 판단 | LLM full read 후 결정 | index + page-summary 만 LLM read | 정확도 유지 + 비용 ↓ |

**Implementation**:
- `work_backlog.md` rewrite → wiki plan R4 anchor 패턴 (`### [[TASK-XXX]] {#task-XXX}`) 적용 (D6)
- `session-start` skill 리팩토링: linear read → index-based
- `MEMORY_GOVERNANCE.md` §2 의 `work_backlog.md` 템플릿 업데이트
- T1 lint 와 결합 (stale/orphan 자동 감지)

**Validation**:
- `[ ]` `work_backlog.md` 모든 entry 가 anchor 부착
- `[ ]` session-start read 시 anchor ID 1:1 lookup 동작 (V-T2a)
- `[ ]` 회귀 test: 10+ task fixture 에서 50%+ read 시간 단축 확인 (V-T2b)

### T3: Memory Ingest (multi-file atomic)  {#t3-memory-ingest}

| Field | Value |
|---|---|
| Statement | session 종료 시 4~6 파일을 atomic 하게 갱신. 부분 write 시 전체 rollback. write lock 으로 동시 ingest 차단 |
| Why | handoff + backlog + tasks + state.json + work_backlog + log 의 partial update 가 의미적 모순을 만듦. memory 는 1 세션 = 1 raw chunk (D9) 이므로 atomic 필수 |
| Implementation | `workflow_writes.ingest_session_atomic()` 신규 — write lock `.ingest_lock` (gitignore, atomic counter), (1) `backlog/YYYY-MM-DD.md` 갱신, (2) `backlog/tasks/TASK-XXX.md` 들 갱신 (frontmatter + outcome), (3) `work_backlog.md` anchor 갱신, (4) `session_handoff.md` 갱신, (5) `state.json` 재생성 (`generate_workflow_state.py` 호출), (6) `memory/log.md` append. 모두 temp dir 에 write → fsync → atomic rename → state.json 만 마지막 (R8·R10 부합) |
| Files | 4~6 동시 갱신: `backlog/YYYY-MM-DD.md` + `backlog/tasks/TASK-XXX.md` (1+) + `work_backlog.md` + `session_handoff.md` + `state.json` + (optional) `memory/log.md` |
| Validator | V-T3a/b/c |

**Validation**:
- `[ ]` 6 파일 갱신 중 부분 실패 = 0건 (atomic 보장, V-T3a)
- `[ ]` `.ingest_lock` 경합 시 reject 로그 (V-T3b)
- `[ ]` state.json 이 모든 파일 path 와 일치 (cross-link lint, V-T3c)

## §6 Anti-Patterns  {#s6-anti-patterns}

| ID | 안티패턴 | 위험 | 대안 |
|---|---|---|---|
| **A5** | `memory/active/` 를 wiki-ingest source 로 사용 | active 는 mutable, freeze 전이라 ingest 중 갱신 시 raw/compiled 모순. R9 위반 | R9: ingest source = `archive/YYYY-MM-DD/` only. active/ 경로 reject |
| **A6** | freeze 없이 archive 로 직접 write (skip session-end trigger) | D8·D9 granularity 무너뜨림. archive 가 "frozen" 의미를 잃음. 다음 세션 raw provenance 추적 불가 | R8: session-end 시에만 freeze. 수동 write 시 lint error |
| **A7** | session 중간에 archive/ 파일 수정 | raw immutability 위반. wiki-ingest source 신뢰성 붕괴 | R8: freeze 후 read-only. 수정 필요 시 next session 에 active/ 에서 갱신 |
| **A8** | `state.json` 와 실제 파일 경로 불일치 | freeze 후 R10 검증 실패 → raw 깨짐. ingest source 로 사용 시 wiki 가 잘못된 provenance 표시 | R10: freeze 직전/직후 state.json path 와 실제 파일 path 자동 검증. 불일치 시 freeze abort |
| **A9** | `work_backlog.md` 를 자유 산문으로 작성 | T2 의 anchor 구조 무너뜨림. R4 위반 (wiki plan). query retrieval 시 O(n) linear read | R4: anchor 기반 구조 강제 (`### [[TASK-XXX]] {#task-XXX}`). T1 lint 가 anchor 미부착 entry 발견 시 reject |

## §10 ADR-005 Draft  {#s10-adr-005-draft}

> **ADR-005: Memory as Raw Layer**
>
> | 필드 | 값 |
> |---|---|
> | Status | proposed |
> | Date | 2026-06-12 |
> | Deciders | Sisyphus, maintainer |
> | Parent | [llm-wiki-convergence-design.md](llm-wiki-convergence-design.md) |
>
> **Context**
>
> Karpathy LLM Wiki 의 3-Layer (raw/wiki/SCHEMA) 와 우리 `ai-workflow/memory/` 의 휘발성 memory 정책이 매핑 가능. 단, Karpathy 의 "immutable raw" 요건을 **시간축** 으로 재해석: session 종료 시 `active/` → `archive/` freeze 가 raw immutability 의 boundary. 현 `memory/` 는 mutable + gitignore 인 채로 운영되어 raw provenance 추적 불가, wiki-ingest source 모호.
>
> **Decision**
>
> memory 에 3-state lifecycle 도입 (active mutable / archive immutable per-session / release deep freeze). wiki-ingest 의 source 는 `archive/` 만. freeze 트랜지션 시 lint 자동 검증. R8~R10 rules 적용.
>
> **Consequences**
>
> Positive:
> - (+) Karpathy 3-Layer 와 1:1 매핑 (raw = archive/, wiki = wiki/)
> - (+) ingest source 명시 (active vs archive) — R9
> - (+) 모순 provenance 추적 가능 (archive session 직접 link)
> - (+) memory + wiki 운영 헌법 단일화 (`MEMORY_GOVERNANCE.md` 확장)
> - (+) session-end 자동 freeze (D8) — manual toil 제거
>
> Negative:
> - (-) freeze 트랜지션 인프라 필요 (skill + lint)
> - (-) 기존 `memory/` → `memory/active/` rename 필요 (P1.5)
> - (-) `work_backlog.md` anchor 전환 (D6) — T2 commit 시점 일괄
> - (-) 6 harness overlay memory/ 동기화 (P4)
>
> **Alternatives considered**
>
> | Option | Reason rejected |
> |---|---|
> | memory 무변경 (방안 B) | ingest source 모호 (active vs archive) — R9 위배로 wiki 와 raw 간 모순 발생 |
> | Git-tracked memory | ADR-001 정책 위반 — Runtime layer 의 gitignore 가 design contract. 영구화 = state/knowledge 경계 붕괴 |
> | Per-task freeze (vs per-session) | granularity 과도 — 1 세션 = 1 raw chunk (D9) 이 분산·merge 의 최소 단위. per-task 면 archive 디렉토리 폭증, lint 비용 ↑ |
> | 수동 freeze (D8 자동 거부) | recall 부재 위험 — 다음 세션 agent 가 freeze 누락 시 raw 손실. 자동화가 단일 source of truth |
> | `memory/` → `memory/raw/` 단일 depth (vs `active/archive/release` 3-depth) | lifecycle 1-step 만 표현. release deep freeze 별도 경로 부재 → wiki 와 archive 가 release 시점 분리 안 됨 |
