## Lesson

### 1 skill = 1 release 분할 정공법

v0.11.20 note 의 "3 beta skill 중 가장 성숙한 것부터 1 release 분할" 정공법 적용. v0.11.21 = robust-patcher (Beta, 가장 성숙) 단독. v0.11.22+ = automated-repro-scaffold / git-conflict-resolver 각각 별도 release.

**lesson (cross-project)**: 동일 deliverable 묶음 안에서 maturity 가 다른 경우 가장 성숙한 것부터 1 release = 1 skill 패턴. v0.11.20 처럼 모두 같은 maturity 면 묶음 OK.

### run_robust_patcher.py 표준화 + atomic semantics

이전 `patch_engine.py` 의 한계:
- 비표준 진입점 (catalog §5 의 `scripts/run_<skill>.py` 패턴 비대칭)
- error_code 비표준 (patch_file_missing / patch_application_failed / patch_engine_runtime_error)
- atomic semantics 미구현 — 실패 시 부분 적용된 상태로 file 변경 가능

v0.11.21 정공법:
- 진입점 표준화 (`run_robust_patcher.py`) + 이전 진입점 trash
- error_code 4종 정합 (missing / malformed / fuzzy / runtime)
- `apply_robust_patch_detailed` 가 fuzzy fail 시 즉시 rollback (원본 보존) — caller 가 partial apply 결과 받지 않음
- per-block detail emit (matched / fuzzy_score / preview) — traceability

**lesson (cross-project)**: idempotent operation (patch / DB transaction / file mutation) 구현 시 **atomic semantics** 필수. 실패 시 caller 가 intermediate state 받지 않도록 rollback. detail emit 은 per-step 으로 분해 (block / tx / operation 단위).

### script 표준화 — `scripts/run_<skill>.py` naming

이전 `patch_engine.py` 는 **skill 의 본질을 설명하지 않는** 이름. v0.11.21 부터 `run_robust_patcher.py` 로 표준화 — `<verb>_<skill_name>.py` 패턴 (dispatcher 가 `python3 run_<skill>.py` 로 직접 호출 가능, catalog §5 의 runtime script column 과 정합).

**lesson (cross-project)**: skill 의 *진입점* script 는 *skill 의 이름* 과 정합해야 함. `engine.py` / `main.py` / `<skill>.py` 같은 추상적 이름 ❌. catalog §5 의 runtime script column 이 "✅ run_<skill>.py" 일 때만 dispatcher 통합 가능.

### 4종 error_code 정합 — trigger / 의미 분리

| code | trigger |
|---|---|
| `missing_required_document` | `--file` 또는 `--patch-file` 경로 부재 (사전 차단) |
| `malformed_patch_block` | patch_content 에 valid SEARCH/REPLACE block 자체가 없음 |
| `fuzzy_match_failed` | SEARCH block 이 fuzzy 0.8 threshold 못 넘김 또는 post-patch SyntaxError |
| `robust_patcher_runtime_error` | 예기치 않은 예외 (catch-all) |

**lesson (cross-project)**: error_code 의 trigger 가 *서로 배타적* 이도록 분리. 동일 error 가 두 code 로 떨어지면 caller 가 분기 처리 곤란. fuzzy_match_failed 의 subcase (fuzzy fail vs SyntaxError) 는 message 로 구분 + 동일 code 유지 — caller 가 fuzzy match 실패 vs syntax 실패 분기는 message 로 가능.
