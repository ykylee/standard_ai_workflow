# Global Workflow Standard

- 문서 목적: 여러 프로젝트에 공통 적용할 수 있는 세션 시작, 작업 기록, 검증, 세션 종료의 최소 공통 규칙을 정의한다.
- 범위: 공통 운영 원칙, 언어/보고 원칙, 컨텍스트 절약 원칙, 상태값, 세션 시작 순서, 작업 기록 최소 필드, 장기 작업 계획 문서, 검증 수준, 결과 기록 원칙, 세션 종료 원칙
- 대상 독자: 개발자, 운영자, AI agent, 프로젝트 관리자
- 상태: draft
- 최종 수정일: 2026-04-18
- 관련 문서: `../templates/project_workflow_profile_template.md`, `../templates/session_handoff_template.md`, `../templates/work_backlog_template.md`

## 1. 공통 원칙

- 새 세션은 항상 현재 상태 요약 문서부터 읽는다.
- 작업은 시작 전에 목적, 범위, 예상 산출물, 영향 문서를 짧게 브리핑한다.
- 작업은 상태 문서에 기록하고, 진행 상태는 `planned`, `in_progress`, `blocked`, `done` 중 하나로 관리한다.
- 날짜별 backlog 는 호스트명과 IP 폴더 아래에 둬 병렬 세션이나 여러 작업 환경이 같은 날짜 파일을 동시에 갱신하지 않도록 한다.
- 코드베이스 분석, 리팩터링, 신규 기능 개발, 마이그레이션처럼 여러 세션에 걸칠 가능성이 큰 작업은 별도 계획 또는 로드맵 문서를 연결한다.
- 검증하지 않은 결과는 완료로 확정하지 않는다.
- 세션 종료 전에는 다음 세션이 바로 이어받을 수 있게 현재 상태를 요약한다.
- 공통 표준은 얇게 유지하고, 프로젝트별 차이는 프로젝트 프로파일에 둔다.

## 1.1 언어와 보고 원칙

- 사용자에게 직접 보여지는 작업 보고, 상태 요약, 문서 초안, handoff, backlog 갱신 문안은 기본적으로 한국어로 작성한다.
- 저장소 표준 문서와 템플릿도 별도 예외가 없으면 한국어를 기본 언어로 유지한다.
- 코드, 명령어, 파일 경로, 설정 key, 외부 시스템 고유 명칭은 필요할 때 원문 그대로 유지할 수 있다.
- 프로젝트 특성상 영어 산출물이 꼭 필요한 경우에는 프로젝트 프로파일에 예외를 명시한다.

## 1.2 컨텍스트 절약 원칙

- 사용자에게 보이지 않는 내부 처리, 중간 분류, 임시 사고 과정은 모델이 가장 효율적인 형태로 수행한다.
- 중간 reasoning 을 장문으로 반복 출력하지 않는다.
- 이미 확인한 사실을 매 단계 길게 재서술하지 않고, 필요한 결론과 다음 행동만 짧게 남긴다.
- 작업 중 누적되는 컨텍스트는 현재 의사결정과 다음 행동에 필요한 정보 중심으로 유지한다.
- 긴 원문 인용, 중복 요약, 불필요한 체크리스트 복제를 피한다.
- 세션 문서에는 최종 결정, 검증 결과, 다음 세션에 필요한 사실만 남기고 내부 탐색 흔적은 최소화한다.
- orchestrator 와 worker 를 나눠 운영할 수 있는 하네스에서는 메인 orchestrator 가 직접 도구 호출을 떠안기보다 task delegation 과 결과 통합에 집중하는 구성을 기본값으로 둔다.
- 실제 탐색, 수정, 검증은 bounded scope worker 에 맡기고, ask 는 genuinely blocking decision 이나 위험한 외부 작업으로만 좁히는 편을 기본 원칙으로 둔다.
- `ai-workflow/` 는 세션 복원과 workflow 상태 관리용 메타 레이어로 보고, 프로젝트 코드/문서 탐색 범위에는 기본적으로 포함하지 않는다.

## 2. 세션 시작 순서

1. 세션 상태 요약 문서를 읽는다.
2. 작업 백로그 인덱스와 최신 날짜 백로그를 읽는다.
3. 진행 중 또는 차단 작업이 있는지 확인한다.
4. 이어서 작업할 항목을 선택하고, 항목에 연결된 계획 또는 로드맵 문서가 있으면 먼저 읽는다.
5. 현재 프로젝트 프로파일을 읽고 저장소별 명령과 문서 구조를 확인한다.

## 3. 작업 상태값

| 상태 | 의미 |
| --- | --- |
| `planned` | 시작 준비는 됐지만 본격 수행 전 |
| `in_progress` | 현재 세션 또는 다음 세션에서 이어서 처리 중 |
| `blocked` | 외부 의존성 또는 결정 대기 때문에 진행 불가 |
| `done` | 완료 기준과 검증 근거를 갖춘 상태 |

## 4. 작업 기록 최소 필드

각 작업 항목은 최소한 아래 필드를 가져야 한다.

- 작업명
- 상태
- 우선순위
- 요청일
- 완료일
- 담당
- 호스트명
- 호스트 IP
- 영향 문서
- 계획 문서
- 작업 내용
- 진행 현황
- 완료 기준
- 작업 결과
- 다음 세션 시작 포인트
- 남은 리스크
- 후속 작업

## 4.1 장기 작업 계획 문서

아래 성격의 작업은 backlog 항목만으로 관리하지 않고 별도 계획 문서를 둔다.

- 코드베이스 분석, 아키텍처 조사, 의존성 정리
- 대규모 리팩터링, 데이터/스키마 마이그레이션, API 계약 변경
- 여러 파일 또는 여러 모듈을 나누어 구현하는 신규 기능
- 하네스, workflow, 배포 자동화처럼 산출물이 여러 문서/스크립트에 흩어지는 작업

계획 문서는 최소한 아래 내용을 포함한다.

- 작업 카테고리와 카테고리 설명
- 배경과 목표
- 범위와 제외 범위
- 카테고리별 확장 기록
- 현재까지 확인한 사실
- 설계 결정과 보류 중인 결정
- 작업 단계와 상태
- 영향 파일/문서
- 검증 계획과 검증 결과
- 다음 세션 시작 포인트
- 남은 리스크

운영 원칙:

- backlog 항목에는 계획 문서 경로를 반드시 연결한다.
- 세션 시작 시 이어서 할 작업을 선택하면 backlog 항목보다 계획 문서를 먼저 읽는다.
- 계획 문서는 장문의 탐색 로그가 아니라, 다시 시작했을 때 낭비와 산출물 충돌을 줄이는 결정/상태 요약이어야 한다.
- 계획 문서 카테고리는 고정 enum 이 아니라 힌트다. 개발, 분석, 리팩터링, 마이그레이션, 운영, 릴리스, 문서, workflow, research 등 프로젝트에 맞는 카테고리를 추가할 수 있다.
- 카테고리별 확장 섹션은 필요한 항목만 유지하고, 새로운 주제가 나오면 섹션을 추가한다.
- 작업이 작고 단일 세션 안에 끝날 가능성이 높으면 계획 문서 대신 backlog 항목만으로 관리할 수 있다.

## 4.2 호스트별 backlog 분리

병렬 작업 환경에서는 같은 날짜의 backlog 파일을 공유하지 않는다.

권장 경로:

- `ai-workflow/project/backlog/<host-name>/<host-ip>/YYYY-MM-DD.md`

운영 원칙:

- `host-name` 과 `host-ip` 는 파일시스템에서 안전한 값으로 정규화한다.
- 같은 날짜라도 호스트 또는 IP 가 다르면 별도 backlog 파일을 쓴다.
- `work_backlog.md` 는 호스트별 날짜 backlog 링크를 모두 인덱싱한다.
- `latest-backlog` 와 `session-start` 는 인덱스 링크를 우선하고, 필요하면 backlog 디렉터리를 재귀적으로 탐색한다.
- 한 호스트에서 시작한 큰 작업을 다른 호스트에서 이어받을 때는 기존 backlog 파일과 연결 계획 문서를 먼저 읽고, 새 호스트 backlog 에 이어받은 사실을 남긴다.

## 5. 검증 수준

검증은 아래 4단계 중 필요한 수준까지 수행한다.

1. 빠른 로컬 검증
2. 격리 검증
3. 실행 확인
4. 결과 기록

문서 변경만 있어도 최소한 정적 무결성 점검은 권장한다.

## 6. 결과 기록 최소 기준

검증을 수행했다면 아래 중 해당되는 결과를 상태 문서에 남겨야 한다.

- 통과한 명령
- 실패한 명령과 원인
- 미실행 항목과 사유
- 실행 확인 요약
- 남은 리스크

## 7. 세션 종료 원칙

- 오늘 작업 결과를 상태 문서에 반영한다.
- 미검증 항목과 남은 리스크를 명시한다.
- 다음 세션 시작 포인트를 짧게 적는다.
- 필요하면 환경 차이 또는 설치 상태도 별도 기록한다.
- 종료 요약은 다음 세션이 바로 이어받는 데 필요한 핵심 사실만 간결하게 남긴다.

## 8. 프로젝트 프로파일과의 관계

이 문서는 공통 코어만 정의한다. 아래 항목은 프로젝트 프로파일로 분리한다.

- 기본 빌드/테스트/실행 명령
- 문서 디렉터리 구조
- 환경 기록 경로
- 프로젝트 특화 검증 포인트
- 병합/승인 예외 규칙
